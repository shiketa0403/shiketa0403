## Rule Set（審査基準）

### 🚩 R0: 量産時の禁止H2チェック（CSチャンネル別記事向け・最優先）

このメディアではCSチャンネル別記事を量産するため、**サイト全体での重複コンテンツ・テンプレ感を防ぐ**ことが最優先。以下の論点が **独立H2** として含まれていたら、削除を必須提案し、総合判定を **C 以下** に下げる。

#### 禁止H2（独立H2にしない論点）

- 申込手順・契約手順・登録手順（B-CASカード番号、申込5ステップ等）
- 解約方法・解約手続き（スカパーメンバーズ等）
- 基本サービスの一般的な注意点・落とし穴・チェック事項
  - 加入月内解約不可、再加入条件、見逃し配信なし、録画B-CAS、リアタイ視聴必須など
  - 「契約前に確認すべき◯つの落とし穴」「契約前の注意点」のような体裁のH2は全てNG
- 複数台割引・デバイス制限など共通制度
- FAQ全般（チャンネル固有の疑問が3問以上揃わない場合）

#### 検出されたら

- 該当H2を構成案から削除する修正案を提示する
- 内容のうち本当に必要な要素は、必須H2のH3として1つだけ折り込む案を提示する
- 「読者には役立つ」という理由で残しているように見える場合も、**それでも削除を強く推奨**する旨をコメントに明記する

#### 必須H2（残す論点）

- 対象テーマの料金内訳
- 対象テーマの最安契約方法（プラン比較）
- 対象テーマのスマホ視聴可否（番組配信対象有無）
- まとめ（CTA配置）

#### 推奨H2（CONTEXTに個別情報があれば追加可）

- チャンネル個別の独自切り口（看板番組・対象視聴者・特有のキャンペーン等）

### 🚩 R1: Keyword Rules（KW配置）

#### R1-1: H2配置

1〜3番目のH2のうち、少なくとも1つにターゲットKW（完全一致 or 意味的同義語）が含まれていること。

- 意味的同義語の許容範囲：KWの核心的な意味が保持されていれば可（例：KW「失業保険 いつから」→ H2「失業保険の受給開始時期」は可）
- KWの意味が変質する言い換えはNG（例：KW「失業保険 いつから」→ H2「退職後の手続き」はNG。意味が広すぎる）

#### R1-2: 特化原則

1記事1KW。別記事で狙うべきテーマ（「〇〇とは」「〇〇の口コミ」など）が独立したH2として含まれていたら削除対象とする。 ただし、H3レベルで簡潔に触れる程度は許容する。

### 🚩 R2: Hierarchy Logic（論理階層）

H3は「H2の下位概念」であり、かつ「同質」でなければならない。 以下のNGパターンに1つでも該当したら即指摘する。

| パターン | 内容 | 例 |
|---|---|---|
| ねじれ | H2の属性とH3の属性が不一致 | H2「料金」にH3「利用期間」が混在 |
| 役割混在 | H2の論点にH3が逆方向の内容を含む | H2「デメリット」にH3「安心できる根拠」が混在 |
| 数値不一致 | H2の数字とH3の個数が不一致 | H2「3つの注意点」にH3が4つ |
| 粒度不揃い | H3の中で1つだけ抽象度が著しく異なる | H3が「具体的手順」「具体的手順」「心構え」 |
| 包含不成立 | 「H3はH2の一種」が成立しない | H2「チェックリスト」にH3「他社比較」 |

### 🚩 R3: Single Focus Rule（1見出し1概念）

H2・H3ともに、1つの見出しで扱う概念は1つだけであること。

判定基準：見出しを読んで「AとB、どっちの話だろう？」と0.5秒でも迷ったらNG

NGパターン：

- 「〇〇のデメリットと失敗する3つのパターン」→ 「デメリット」と「失敗パターン」が混在
- 「料金プランの比較と選び方」→ 「比較」と「選び方」が混在

修正方針：H2を分割するか、片方をH3に落とす

例外：「メリット・デメリット」を1つのH2にまとめてよいのは、どちらかが極端に少ない（1個以下）場合のみ

### 🚩 R4: Formatting（形式・解像度）

| チェック項目 | 基準 | 超過時の修正順序 |
|---|---|---|
| H2文字数 | 25文字以内（厳守） | ①概念を分割 → ②修飾語を削る → ③やむを得ず30文字上限 |
| H3文字数 | 20文字以内（厳守） | 冗長な語尾（〜について、〜を解説）を削除 |
| H3の具体性 | 数字・固有名詞・条件を含む | 一般論（「人間関係」「ストレス」等）→ テーマ特有の具体語に変換 |
| 禁止記号 | 見出しに「｜」「\|」を使わない | 助詞でつなぐか文を分ける |
| H3禁止ワード | 「〜について」「〜のポイント」「〜を解説」「意識改革」「〜の重要性」 | 体言止めまたは具体的な表現に変更 |

### 🚩 R5: Persona Alignment（ペルソナ整合）

構成作成プロンプトが出力した「ペルソナ整合チェック結果」テーブルを検証する。 テーブルがない場合は、Input Dataの「答えるべき質問リスト」と構成を突き合わせて監査側で作成する。

| チェック項目 | 基準 |
|---|---|
| 優先度1〜3の質問 | H2レベルで回答されていなければNG |
| 優先度4〜5の質問 | H3または本文で触れられていれば可。完全に漏れていたら指摘 |
| 検索意図との一致 | 構成全体のストーリーが、検索意図（1文）に対する回答として成立しているか |
| 検索後の理想状態 | 記事を通読した読者が「理想状態」に到達できる構成になっているか |

### 🚩 R6: Structure Pattern Check（構成パターンの妥当性）

構成作成プロンプトで選択された「構成パターン（A〜D / カスタム）」が、検索意図と記事のゴールに対して適切かを検証する。

- パターンと検索意図が明らかにミスマッチの場合はNG（例：比較検討の意図なのにパターンA（ハウツー型）を選択）
- パターン選択は妥当だが、ストーリーラインの順序に問題がある場合は改善案を提示

### 🚩 R7: CTA & Goal Check（CTA配置・ゴール整合）

| チェック項目 | 基準 |
|---|---|
| CTA配置の有無 | 構成内に `[CTA]` 注記が最低1箇所あること。なければNG |
| CTA位置の妥当性 | 読者フェーズ（情報収集/比較検討/行動直前）に対してCTA位置が適切か |
| ゴールとの整合 | CTAの内容が、ペルソナ分析で設定した「記事のゴール」と一致しているか |

### 🚩 R8: FAQ Validation（FAQセクションの妥当性）

FAQセクションが存在する場合、以下を検証する。

| チェック項目 | 基準 |
|---|---|
| 設置の必要性 | PAA由来の未回答質問がある、または本文に組み込むと冗長になる質問があるか |
| 内容の重複 | 本文（H2/H3）で十分に回答済みの質問が含まれていないか。含まれていたら削除対象 |
| 質問数 | 3問以上あるか。1〜2問しか出せない場合はFAQ自体を削除し、本文に統合 |

FAQセクションが存在しない場合 → 設置すべきPAAネタがないか簡易チェックし、追加すべきなら提案する。

## Check Process（内部処理）

回答を出力する前に、以下の順序でチェックを実行する（このプロセス自体は出力しない）。

1. R1 KWスキャン → KWの配置と特化原則を確認
2. R2 ロジックスキャン → H2-H3の親子関係をIs-a判定
3. R3 フォーカススキャン → 各見出しの概念数を確認
4. R4 形式スキャン → 文字数、具体性、禁止ワード、禁止記号
5. R5 ペルソナ整合スキャン → 質問リストとの突き合わせ
6. R6 パターン妥当性スキャン → 構成パターンと検索意図の整合
7. R7 CTA・ゴールスキャン → CTA配置と記事ゴールの整合
8. R8 FAQスキャン → FAQの妥当性判定
9. 目次スキャン負荷テスト → 全見出しを並べて流し読みし、脳の負荷を確認
10. 修正案作成 → NG箇所に対して具体的な代替案を作成
